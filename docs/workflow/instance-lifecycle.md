# 流程实例生命周期

流程实例从发起到结束，经历完整的生命周期。了解生命周期有助于理解流程的运行机制，以及如何处理各种边界情况。

## 生命周期图

```
    ┌─────────┐   提交草稿   ┌──────────────┐   全部通过   ┌──────────┐
    │  草稿   │ ───────────→ │   运行中     │ ───────────→ │  已通过  │
    └─────────┘              └──────────────┘              └──────────┘
                                   │
                         ┌─────────┼─────────┐
                         ↓         ↓         ↓
                   ┌──────────┐ ┌────────┐ ┌────────┐
                   │  已驳回  │ │ 已撤回 │ │ 已取消 │
                   └──────────┘ └────────┘ └────────┘
                         │         │
                         └──────重新提交为新草稿──────→ 草稿
```

## 状态说明

| 状态 | 说明 | 触发条件 |
| --- | --- | --- |
| `draft` 草稿 | 已保存但未提交流转的流程实例 | 发起人保存草稿 |
| `running` 运行中 | 流程正在执行中 | 用户成功发起申请或提交草稿 |
| `approved` 已通过 | 所有节点审批通过，流程完成 | 最后一个节点完成后无后续节点 |
| `rejected` 已驳回 | 某个节点被驳回，流程终止 | 审批节点驳回且策略为终止流程 |
| `withdrawn` 已撤回 | 发起人主动撤回 | 发起人在流程运行中点击撤回 |
| `cancelled` 已取消 | 管理员在流程监控中取消运行中的实例 | 拥有 `workflow:instance:cancel` 的管理员取消 |

## 发起流程

### 入口

用户进入「工作流 → 发起工作台」，选择已发布的流程定义，即可发起申请；也可以保存为草稿后在「我的申请」继续编辑并提交。

### 发起条件

- 流程定义状态必须为「已发布」
- 当前用户必须在流程定义的「发起人范围」内（详见 [权限与范围控制](./permissions.md)）

### 发起参数

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| definitionId | 是 | 流程定义 ID |
| title | 是 | 申请标题 |
| formData | 否 | 表单数据，键值对对象 |
| asDraft | 否 | 是否保存为草稿 |

### 发起后发生的事件

1. 系统校验流程定义有效性和用户权限
2. 创建流程实例记录
3. 提交申请时生成业务编号并创建首批任务；保存草稿时不进入流转
4. 提交申请触发 `instance.created` 事件
5. 触发各任务的 `task.created` 和 `task.assigned` 事件
6. 提交申请后实例状态为 `running`，保存草稿后实例状态为 `draft`

草稿实例可继续编辑和提交。被驳回或已撤回的实例可重新提交，系统基于原表单数据创建新的草稿。

## 审批任务

### 待审批入口

审批人进入「工作流 → 待我审批」，可看到所有待处理的任务。

### 审批操作

**通过**：
- 审批人填写意见（可选）后点击「通过」
- 系统根据审批方式（会签/或签/依次审批/比例会签/随机一人/自动通过）判断是否推进流程
- 如果是或签且已通过，同节点的其他待审批任务自动标记为「已跳过」
- 如果是会签，等待所有审批人通过后才推进
- 如果是依次审批，等待当前人通过后才会给下一人生成任务

**驳回**：
- 审批人填写驳回意见后点击「驳回」
- 根据节点配置的驳回策略执行不同操作：

| 驳回策略 | 行为 |
| --- | --- |
| 终止流程 | 实例状态变为「已驳回」，流程结束 |
| 退回上一步 | 实例回到上一个审批/办理节点，重新生成任务 |
| 退回发起人 | 实例回到发起人节点，重新发起 |
| 退回指定节点 | 实例回到指定的节点（通过节点标识 `key` 引用），重新生成任务 |

### 任务状态流转

```
待处理 (pending)
    ├── 通过 → 已通过 (approved)
    ├── 驳回 → 已驳回 (rejected)
    └── 流程退回/撤回 → 已跳过 (skipped)

等待中 (waiting)
    └── 满足条件后 → 待处理 (pending) 或自动完成
        · 顺序会签：前序人员通过后转 pending
        · 延迟器：到达唤醒时间后自动推进
        · 触发器(callback) / 外部审批：收到回调后推进
        · 子流程：子实例结束后回写并推进父流程
```

## 自动节点

除人工审批/办理节点外，流程还支持以下自动推进的节点，它们由引擎或后台调度器驱动，无需人工操作：

| 节点 | 行为 |
| --- | --- |
| 延迟器（delay） | 创建 `waiting` 任务，由调度器在「固定时长 / 指定日期」到达时唤醒并推进 |
| 触发器（trigger） | 进入节点时执行 HTTP 调用或数据读写；`callback` 类型会等待外部回调后再推进，详见 [触发器节点](./trigger-nodes.md) |
| 抄送（ccNode） | 解析接收人后即时推进，不阻塞流程 |
| 子流程（subProcess） | 发起子流程实例；同步模式等待子实例结束后回写出参并推进，支持多实例（并行/串行）汇聚，详见 [节点配置指南](./node-config.md#子流程) |

### 审批超时

审批/办理节点可配置超时处理：到达超时时间后由后台定时任务执行「自动提醒 / 自动通过 / 自动拒绝」，提醒次数耗尽后还可执行升级动作（转交上级等）。详见 [节点配置指南](./node-config.md)。

## 撤回流程

### 入口

发起人进入「工作流 → 我的申请」，找到运行中的流程，点击「撤回」。

### 撤回条件

- 当前用户必须是发起人
- 流程状态必须为「运行中」
- 流程全局设置允许撤回

### 撤回后

1. 实例状态变为「已撤回」
2. 所有未处理的任务自动标记为「已跳过」
3. 触发 `instance.withdrawn` 事件

## 流程完成

当最后一个审批/办理节点完成，且没有后续节点时，流程实例状态变为「已通过」。

### 完成条件

- 所有节点任务都已处理（通过或跳过）
- 没有未执行的分支

### 完成后

- 实例状态为「已通过」
- 触发 `instance.approved` 事件
- 所有任务已处理完毕

## 异常处理

### 审批人解析为空

当审批人节点配置的审批人解析为空时（如部门负责人不存在、表单字段为空），根据节点的「空审批人处理策略」自动处理：

| 策略 | 结果 |
| --- | --- |
| 自动通过 | 节点自动通过，流程继续 |
| 转交管理员 | 任务转交给系统管理员 |
| 自动拒绝 | 节点自动拒绝，流程按驳回策略处理 |
| 转交指定人 | 任务转交给预设人员 |

### 外部审批派发失败

当启用外部审批的审批节点派发失败时（网络错误、超时、非 2xx 响应），根据节点的 `fallbackStrategy` 处理：

| 策略 | 结果 |
| --- | --- |
| 人工处理（默认） | 任务保持待人工审批状态 |
| 自动通过 | 任务自动通过 |
| 自动拒绝 | 任务自动拒绝 |

## 实例查询

### 发起工作台

用户选择自己有权限发起的已发布流程，填写表单后提交或保存草稿。

### 我的申请

用户查看自己发起的所有流程实例，支持按状态筛选（草稿/运行中/已通过/已驳回/已撤回/已取消），并可继续编辑草稿、撤回、催办、补加抄送。

### 待我审批

用户查看需要自己审批的所有待处理任务，点击可直接进入审批页面。

### 抄送我的 / 我已办

用户查看抄送给自己的流程任务，以及自己已经处理过的审批/办理任务。

### 流程监控（管理员）

管理员可查看全局所有流程实例，支持：
- 按状态统计（运行中/已通过/已驳回/已撤回/已取消的数量）
- 按流程名称、发起人、分类筛选
- 查看实例详情和任务流转记录
- 查看超时预警、导出实例列表
- 取消运行中的实例、删除已结束的实例（需对应权限）
- 强制跳转到指定节点、改派待办任务
